一、类加载机制

类加载过程

加载 >> 验证 >> 准备 >> 解析 >> 初始化 >> 使用 >> 卸载

  • 加载：在硬盘上查找并通过IO读入字节码文件，使用到类时才会加载，例如调用类的main()方法，new对象等等，在加载阶段会在内存中生成一个代表这个类的java.lang.Class对象，作为方法区这个类的各种数据的访问入口

  • 验证：校验字节码文件的正确性

  • 准备：给类的静态变量分配内存，并赋予默认值

  • 解析：将符号引用替换为直接引用，该阶段会把一些静态方法(符号引用，比如main()方法)替换为指向数据所存内存的指针或句柄等(直接引用)，这是所谓的静态链接过程(类加载期间完成)，动态链接是在程序运行期间完成的将符号引用替换为直接引用

  • 初始化：对类的静态变量初始化为指定的值，执行静态代码块

  • 卸载：

    如何判断一个类是无用的类

    方法区主要回收的是无用的类，那么如何判断一个类是无用的类呢？

    类需要同时满足下面3个条件才能算是 “无用的类” ：

    • 该类所有的对象实例都已经被回收，也就是 Java 堆中不存在该类的任何实例。
    • 加载该类的 ClassLoader 已经被回收。
    • 该类对应的 java.lang.Class 对象没有在任何地方被引用，无法在任何地方通过反射访问该类的方法。

类加载器和双亲委派机制

类加载器

  • 引导类（启动）加载器（Bootsrap ClassLoader） ：负责加载支撑JVM运行的位于JRE的lib目录下的核心类库，比如rt.jar、charsets.jar等

  • 扩展类加载器（Extension ClassLoader）：负责加载支撑JVM运行的位于JRE的lib目录下的ext扩展目录中的JAR类包

  • 应用程序类加载器（Application ClassLoader）：负责加载ClassPath路径下的类包，主要就是加载你自己写的那些类

  • 自定义加载器：负责加载用户自定义路径下的类包

自定义加载器

自定义类加载器只需要继承 java.lang.ClassLoader 类

重写 findClass 空方法，打破双亲委派需重写 loadClass 实现自己的加载逻辑，不委派给双亲加载

tomcat打破双亲委派

jsp，freemarker热加载，每次修改不需要重启服务

三、对象创建与内存分配

对象的创建过程

  1. 类加载检查 ： 虚拟机遇到一条new指令时，首先将去检查这个指令的参数是否能在常量池中定位到一个类的符号引用，并且检查这个符号引用代表的类是否已被加载、解析和初始化过。如果没有，那必须先执行相应的类加载过程

  2. 分配内存：对象所需内存的大小在类 加载完成后便可完全确定，为对象分配空间的任务等同于把 一块确定大小的内存从Java堆中划分出来

  3. 初始化：对所有非静态成员变量进行默认初始化

  4. 设置对象头：虚拟机要对对象进行必要的设置，例如这个对象是哪个类的实例、如何才能找到类的元数据信息、对象的哈希码、对象的GC分代年龄等信息。这些信息存放在对象的对象头Object Header之中。

对象内存分配

对象内存分配流程

、

逃逸分析：为了减少临时对象在堆内分配的数量，JVM通过逃逸分析对象是否会被外部访问，如没有则直接到栈上分配内存，内存空间就可以随栈帧出栈而销毁，就减轻了垃圾回收的压力

Minor GC/Young GC：指发生新生代的的垃圾收集动作，Minor GC非常频繁，回收速度一般也比较快。eden区满了后会触发minor gc，剩余存活的对象会被挪到为空的那块survivor区

Major GC/Full GC：一般会回收老年代 ，年轻代，方法区的垃圾，Major GC的速度一般会比Minor GC的慢10倍以上。移动到老年代对象可用空间不可用时，那么会触发full gc，full gc完之后如果还是没有空间放minor gc之后的存活对象，则也会发生“OOM”

四、垃圾收集器与对象内存回收

堆中对象回收前需判断对象是否是垃圾对象，即没有被任何使用的对象。

可回收对象的判断方法

  • 引用计数法：给对象中添加一个引用计数器，每当有一个地方引用它，计数器就加1；当引用失效，计数器就减1；任何时候计数器为0的对象就是不可能再被使用的。这个方法实现简单，效率高，但是目前主流的虚拟机中并没有选择这个算法来管理内存，其最主要的原因是它很难解决对象之间相互循环引用的问题

  • 可达性分析算法：将“GC Roots” 对象作为起点，从这些节点开始向下搜索引用的对象，找到的对象都标记为非垃圾对象，其余未标记的对象都是垃圾对象。真正宣告一个对象死亡，至少需要标记两次

    GC Roots根节点：线程栈的本地变量、静态变量、本地方法栈的变量等等

​ 常见引用类型：强引用、软引用、弱引用、虚引用

​ finalize()方法：如果这个对象覆盖了finalize方法，对象被回收前会执行此方法

垃圾收集器

垃圾收集算法：

  • 分代收集理论
  • 新生代：复制算法
  • 老年代：标记整理算法、标记清除算法

垃圾收集器：

  • Serial(串行)收集器(-XX:+UseSerialGC -XX:+UseSerialOldGC)

    新生代采用复制算法，老年代采用标记-整理算法。单线程收集器，STW时间长，用户体验不好

  • Parallel Scavenge收集器(-XX:+UseParallelGC(年轻代),-XX:+UseParallelOldGC(老年代))

    新生代采用复制算法，老年代采用标记-整理算法。Serial收集器的多线程版本，JDK8默认的新生代和老年代收集器

  • ParNew收集器(-XX:+UseParNewGC)

    新生代采用复制算法，老年代采用标记-整理算法。跟Parallel收集器很类似，区别主要在于它可以和CMS收集器配合使用

  • CMS收集器(-XX:+UseConcMarkSweepGC(old))

    与ParNew收集器配合使用，新生代使用ParNew，老年代使用CMS，并发收集、低停顿。缺点，吃CPU资源；清除收集产生大量内存碎片，需配合 -XX:+UseCMSCompactAtFullCollection使用；无法处理浮动垃圾(在并发标记和并发清理阶段又产生垃圾，这种浮动垃圾只能等到下一次gc再清理了）

    G1收集器(-XX:+UseG1GC)

    ZGC收集器(-XX:+UseZGC)

    如何选择垃圾收集器

    1. 优先调整堆的大小让服务器自己来选择

    2. 如果内存小于100M，使用串行收集器

    3. 如果是单核，并且没有停顿时间的要求，串行或JVM自己选择

    4. 如果允许停顿时间超过1秒，选择并行或者JVM自己选

    5. 如果响应时间最重要，并且不能超过1秒，使用并发收集器

    6. 堆内存4G以下可以用parallel，4-8G可以用ParNew+CMS，8G以上可以用G1，几百G以上用ZGC

      JDK 1.8默认使用 Parallel(年轻代和老年代都是)

      JDK 1.9默认使用 G1

五、JVM调优

查看应用程序的线程id：jps

查看堆信息 jmap

  • 查看堆内存实例数：
    • 查看历史生成的实例： jmap -histo <pid> > ./log.txt
    • 查看当前存活的实例：jmap -histo:live <pid> > ./log.txt
  • 导出堆内存信息文件，载入分析工具进行分析：jmap -dump:format=b,file=eureka.hprof <pid>
    • 配置内存溢出自动导出：-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=./

查看线程信息 jstack

  • 查看死锁或高CPU：jstack -F <pid>

查看jvm参数 jinfo

  • jinfo -flags <pid>

查看堆内存各部分的使用量 jstat

  • jstat -gc <pid> [打印间隔] [打印次数]

    通过查看GC发生的次数，时间间隔来决定是否优化

    S0C：第一个幸存区大小；S1C：第二个幸存区的大小；EC：伊甸园区的大小；OGCMN：老年代最小容量

    OGCMX：老年代最大容量；OGC：当前老年代大小；OC:当前老年代大小；MCMN:最小元数据容量

    MCMX：最大元数据容量；MC：当前元数据空间大小；CCSMN：最小压缩类空间大小；CCSMX：最大压缩类空间大小

    CCSC：当前压缩类空间大小‘YGC：年轻代gc次数’FGC：老年代GC次数

调优工具：阿里巴巴Arthas https://alibaba.github.io/arthas/
